# Deployment Recommendations

Этот файл фиксирует текущие рекомендации по хостингу и системным требованиям для `EnglishBot`.

Документ намеренно считается временным и должен обновляться по мере развития продукта, архитектуры и внешних интеграций.

## Current Technical Context

По состоянию на сейчас репозиторий предполагает такой runtime-профиль:

- Telegram-бот работает как долгоживущий процесс через polling
- runtime-состояние хранится в локальной SQLite базе
- изображения и review-ассеты сохраняются в локальную файловую систему
- редакторские флоу могут использовать локальный `Ollama`
- генерация изображений может использовать локальный `ComfyUI`
- `Pixabay` используется как внешняя HTTP-интеграция для поиска изображений

Из этого следует, что текущая версия проекта лучше всего подходит для запуска на одном постоянном сервере с persistent storage.

## Recommended Hosting Options

### 1. Hetzner Cloud

Основная рекомендация для текущей стадии проекта.

Почему подходит:

- хороший баланс цены и ресурсов
- удобно для одного долгоживущего Python-процесса
- естественно подходит под SQLite и локальные `assets`
- удобно держать `systemd` или Docker-контейнер
- достаточно просто добавить отдельный volume или регулярные backup-скрипты

Когда выбирать:

- нужен недорогой и понятный VPS
- бот работает на одном инстансе
- хочется минимально сложную инфраструктуру

### 2. DigitalOcean Droplet

Хороший запасной вариант.

Почему подходит:

- простой UX и понятная инфраструктура
- нормален для Python-бота с локальным диском
- удобно стартовать без сложной платформенной адаптации

Когда выбирать:

- важнее удобство панели и привычный managed UX
- цена не критична

### 3. Render / Railway / Fly.io

Возможны, но сейчас не являются лучшим первым выбором.

Почему с ними сложнее:

- проект опирается на локальную SQLite базу и локальные файлы
- для таких платформ обычно удобнее stateless-сервисы или внешняя managed storage-схема
- при дальнейшем росте все равно, скорее всего, придется выносить БД и файловое хранение

Когда выбирать:

- если вы готовы адаптировать архитектуру под platform-style deployment
- если позже планируется уход от локальной SQLite и локальных `assets`

### 4. Runpod

Не как основная площадка для бота, а как отдельный слой для GPU-задач.

Когда полезен:

- если `ComfyUI` нужно вынести в отдельный GPU-сервис
- если генерация изображений станет регулярной и тяжелой

## Machine Requirements

### Scenario A: Only Bot Runtime

Состав:

- Telegram bot
- SQLite
- локальные `assets`
- без локального `Ollama`
- без локального `ComfyUI`

Минимум:

- `1 vCPU`
- `2 GB RAM`
- `20 GB SSD`

Комфортно:

- `2 vCPU`
- `4 GB RAM`
- `40 GB SSD`

### Scenario B: Bot + Editor Flows + External Integrations

Состав:

- Telegram bot
- SQLite
- локальные `assets`
- `Pixabay`
- без локального AI на этой же машине

Рекомендация:

- `2 vCPU`
- `4 GB RAM`
- `40-60 GB SSD`

Это выглядит как лучший базовый production-вариант на текущем этапе.

### Scenario C: Bot + Local Ollama

Состав:

- весь runtime бота
- локальный `Ollama`
- модель уровня `qwen2.5:7b`

Минимум:

- `4 vCPU`
- `8 GB RAM`
- `60 GB SSD`

Комфортно:

- `8 vCPU`
- `16 GB RAM`
- `80-100 GB SSD`

Примечание:

- для локального LLM лучше сразу закладывать запас по RAM и диску
- если импорт используется редко, может быть выгоднее вынести LLM в отдельный сервис

### Scenario D: Bot + Local ComfyUI

Состав:

- весь runtime бота
- локальный `ComfyUI`

Рекомендация:

- `8 vCPU`
- `16-32 GB RAM`
- `100 GB+ SSD`
- `NVIDIA GPU` с `12 GB+ VRAM`

Практический вывод:

- `ComfyUI` лучше держать отдельно от основного бота
- сам бот можно оставить на обычном VPS
- GPU-генерацию лучше выносить в отдельный runtime или отдельную машину

## Current Recommended Deployment Shape

Для текущего состояния репозитория рекомендуемая схема такая:

- один VPS
- Ubuntu `22.04` или `24.04`
- Python `3.11`
- запуск через `systemd` или Docker
- persistent storage для `data/` и `assets/`
- отдельные backup-правила для `data/englishbot.db` и каталога `assets/`

### Workbook Import Recovery

Для workbook import теперь есть встроенный safety-path:

- перед применением workbook создаётся pre-import snapshot SQLite базы
- backup сохраняется в `data/backups/`
- изменения по темам применяются атомарно в одной SQLite транзакции

Практически это означает:

- если import падает до commit, SQLite откатывает изменения автоматически
- если import завершился, но результат оказался неверным, можно вручную восстановить последний pre-import backup

Минимальный rollback-порядок:

1. остановить бота
2. выбрать нужный backup в `data/backups/`
3. заменить `data/englishbot.db`
4. снова запустить бота

Если import запускался через CLI, путь к backup выводится в результате команды.

Предпочтительный практический вариант:

- хостинг: `Hetzner Cloud`
- размер: `2 vCPU / 4 GB RAM / 40 GB SSD`
- локально запускать только бота
- `Ollama` и `ComfyUI` не размещать на той же машине, пока это не стало реально необходимым

## Why Serverless Is Not The Best Fit Yet

Текущая архитектура пока не очень подходит для serverless-модели, потому что:

- бот запущен как постоянный polling-процесс
- runtime хранится в SQLite
- картинки и промежуточные review-ассеты лежат на локальном диске
- часть интеграций ожидает локальные HTTP-сервисы рядом с ботом

Если позже появятся:

- webhook-режим вместо polling
- внешняя БД
- объектное хранилище вместо локальных файлов
- выделенные AI-сервисы

тогда можно будет пересмотреть выбор хостинга в сторону более platform-native вариантов.

## Next Revision Triggers

Этот документ стоит обновить, когда будет сделано хотя бы одно из следующего:

- переход с SQLite на Postgres
- переход с polling на webhook
- вынос `assets` в S3-совместимое хранилище
- вынос `Ollama` в отдельный сервис
- вынос `ComfyUI` в отдельный GPU-runtime
- появление фоновых очередей или планировщика задач
- рост нагрузки до нескольких параллельных editor/learner flows
