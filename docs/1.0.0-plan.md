# 1.0.0 Plan

This document fixes the intended scope of `1.0.0` and records the intended
release surface for the cleaned-up repository.

The main goal is not to keep refactoring forever. The goal is to ship a small,
teachable, stable version of the project with a clear core product boundary.

## Product Goal

`1.0.0` should mean:

- the bot has one obvious runtime: `python -m englishbot`
- the default devcontainer does not require local AI services
- the core product is easy to explain to a new developer
- the project can be used as a teaching example without dragging every optional
  tool into the main story

## What Is In 1.0.0

The core product for `1.0.0` is:

- Telegram learner flows
- words and lessons
- homework
- per-user learning progress
- learner-facing summaries and simple stats
- lightweight teacher or parent administration inside the same bot runtime
- SQLite as the local runtime store

In architecture terms, the main story should read as:

1. content exists in SQLite
2. the learner chooses or receives a study path
3. the bot runs a study session
4. answers update progress
5. homework and progress screens reflect that state

## What Is Optional In 1.0.0

These parts remain supported, but they are not the core story of the version:

- raw-text import tooling
- image generation
- image rerank and image review tooling
- TTS runtime and audio batch tools
- Telegram Web App administration
- local Ollama and ComfyUI development services

Important rule:

- optional tooling must not be required for the default learner bot startup path

## What A Shim Is

A `shim` is a temporary compatibility module.

It exists so that code can move to a better package location without breaking:

- old imports
- tests
- local scripts
- ongoing refactors

Typical shim lifecycle:

1. move the real implementation to the correct module
2. keep the old module as a thin re-export shim
3. gradually update internal imports to the new location
4. remove the shim in a later cleanup wave

Example pattern:

- real code lives in `src/englishbot/telegram/tts.py`
- old `src/englishbot/telegram_tts.py` would remain as a thin compatibility file
  during a transition

## Internal Shim Cleanup Status

The internal transition shims used during the refactor have been removed before
`1.0.0`.

The repository no longer keeps root-level compatibility modules for:

- runtime wiring such as `bootstrap.py` or `ollama_runtime.py`
- presentation helper modules such as `bot_*_ui.py`
- historical top-level `telegram_*.py` handler or helper modules

The final internal module layout now points directly to:

- `src/englishbot/application/training_runtime.py`
- `src/englishbot/importing/runtime.py`
- `src/englishbot/importing/ollama_runtime.py`
- `src/englishbot/telegram/...`
- `src/englishbot/presentation/...`

Top-level CLI entrypoints such as `python -m englishbot.import_lesson_text`
remain on purpose. They are part of the public command surface, not internal
compatibility shims.

## Release Checklist

The repository is ready for `1.0.0` only when the following are true:

- `python -m englishbot` is the documented default runtime
- default devcontainer works without Ollama and ComfyUI
- default `docker compose up -d --build` starts only the bot runtime
- optional Docker services such as Web App, TTS, and nginx live in an explicit optional compose overlay
- README quickstart matches the actual runtime
- core vs optional functionality is documented clearly
- no required product behavior depends on optional AI services
- the test suite is green
- the current architecture docs point to the real module layout

## Current Recommended Next Step

The next step is not another broad module move by default.

The recommended order is:

1. treat the current layout as the `1.0.0` candidate architecture
2. document the boundaries and release surface clearly
3. decide whether `src/englishbot/bot.py` needs one more meaningful reduction or
   can be frozen as the transitional facade for the release
4. prepare release notes and version metadata

## Non-Goals For 1.0.0

The following are intentionally out of scope:

- microservices
- event buses or brokers
- CQRS or event sourcing
- a full multi-tenant platform redesign
- a full sense-level dictionary model
- a large frontend rewrite

The point of `1.0.0` is clarity and stability, not maximum abstraction.

Current release-fixation checkpoint:

- the default devcontainer is the lightweight default profile, with `cpu` and `gpu` as explicit AI profiles
- the repository has a first release changelog entry in [CHANGELOG.md](/workspaces/EnglishBot/CHANGELOG.md)
